# value-compass

## Structure

```
value-compass/
  backend/
    api/
    poller/
    db/
    common/
    infra/
      systemd/
    requirements.txt
  app/
    VCA.xcodeproj
    Sources/
      App/
        AppFeature/      # TCA @Reducer features (root, navigation, screens)
        Dependencies/    # @DependencyClient interfaces
      Features/          # SwiftUI views wrapping TCA Stores
      DesignSystem/      # Typography, semantic tokens
      Assets/            # Asset catalog and app icon
      Backend/
        Contracts/       # Protocols App/Features depend on
        Models/          # Domain & SwiftData models
        Networking/      # SwiftOpenAPIGenerator output, API client, version monitor
        Services/        # Local computation engines (e.g. ContributionCalculator)
    Tests/
      VCATests/
    build.sh
    run.sh
  .github/
    workflows/
  openapi.json
  README.md
```

## Ownership

| Directory | Purpose |
|---|---|
| `backend/api` | FastAPI application |
| `backend/poller` | APScheduler nightly job |
| `backend/db` | Alembic migrations, shared models |
| `backend/common` | Shared config, constants, utilities |
| `backend/infra/systemd` | `vca-api.service`, `vca-poller.service` |
| `app/Sources/App` | App entry, lifecycle, persistence + design-system bootstrap |
| `app/Sources/App/AppFeature` | TCA `@Reducer` features (root, navigation, screen reducers) |
| `app/Sources/App/Dependencies` | `@DependencyClient` interfaces injected into reducers |
| `app/Sources/Features` | SwiftUI views that wrap TCA `Store`s |
| `app/Sources/DesignSystem` | Typography, semantic color tokens |
| `app/Sources/Assets` | Asset catalog and app icon |
| `app/Sources/Backend/Contracts` | Protocols App/Features may depend on |
| `app/Sources/Backend/Models` | Domain and SwiftData models |
| `app/Sources/Backend/Networking` | API client, version monitor, Keychain glue, SwiftOpenAPIGenerator output |
| `app/Sources/Backend/Services` | Local computation engines (e.g. `ContributionCalculator`) |
| `.github/workflows` | CI/CD pipelines |
| `openapi.json` | Auto-generated by FastAPI, consumed by SwiftOpenAPIGenerator |

## Principles

- Monorepo — backend, iOS, and infra in one repository.
- iOS app architecture is The Composable Architecture (`swift-composable-architecture` 1.25.5); reducers live in `app/Sources/App/AppFeature/` and dependency-injection interfaces in `app/Sources/App/Dependencies/`.
- `openapi.json` is the contract between backend and iOS, never edited manually.
- `backend/common` and `backend/db` are shared between `api` and `poller` — they never drift.
- Generated Swift networking code lives in `app/Sources/Backend/Networking/`, never edited manually.

## CI/CD

CI runs on git events. Apple deployment to App Store Connect is manually triggered for MVP; backend deploys require no manual SSH after the initial VM setup. Workflows live in `.github/workflows/`:

| Workflow             | Trigger                                  | Purpose                                                |
|----------------------|------------------------------------------|--------------------------------------------------------|
| `backend-ci.yml`     | PR / push to `main` touching `backend/`  | Install deps, compile-check, run tests.                |
| `backend-deploy.yml` | Push to `main` or `backend-v*` tag       | SSH to Azure VM, pull, install, **migrate**, restart, health-check, tag. |
| `vc-services-AutoDeployTrigger-*.yml` | Manual `workflow_dispatch` only | Build and deploy the FastAPI Container Apps service from `backend/api`. |
| `ios-ci.yml`         | PR / push to `main` touching `app/`           | `xcodebuild` on macOS (no signing).                    |
| `ios-deploy.yml`     | Manual `workflow_dispatch` with target ref | Sign, archive, export, upload to App Store Connect.    |

Backend runtime secrets (`SUPABASE_URL`, `POLYGON_API_KEY`, `APNS_CERT`)
are not consumed by the workflow itself — they are read by the systemd
units on the VM and are set during the one-time VM bootstrap.

### Deploy invariants

1. Database migrations (`alembic upgrade head`) **always** run before
   `vca-api` / `vca-poller` are restarted.
2. The `/health` endpoint must return `200` before a release tag is
   created — a failed health check fails the workflow.
3. Secrets are never stored in code; only GitHub Secrets.

### Rollback

Re-tag a previous good commit with a new release tag and push it:

```sh
git tag backend-v$(date -u +%Y%m%d-%H%M%S) <previous-good-sha>
git push origin backend-v<...>
```

For iOS rollback, manually run `ios-deploy.yml` from GitHub Actions and set `ref` to the previous good commit SHA, branch, or tag.

### TestFlight setup

The iOS deploy workflow uses manual signing for repeatable CI uploads. Complete the Apple Developer, App Store Connect, and GitHub secret checklist in `docs/testflight-readiness.md` before running `ios-deploy.yml`.

### Initial VM setup (one time, manual)

1. Provision Azure VM (B1s) via the Azure Portal.
2. SSH in.
3. Install Python 3.11+.
4. Clone the repo to `/opt/value-compass` and create the venv at
   `/opt/value-compass/.venv`.
5. `pip install -r backend/requirements.txt`.
6. Symlink the units in `backend/infra/systemd/` into `/etc/systemd/system/`
   and `systemctl enable` them.
7. Set environment variables (from GitHub Secrets) for the systemd
   units (e.g. via `/etc/systemd/system/vca-api.service.d/override.conf`).
8. Configure the Azure firewall to allow port 443 from Cloudflare IP
   ranges only.
9. All subsequent deploys are automated by `backend-deploy.yml`.
